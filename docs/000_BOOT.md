## REQUIRED VERIFICATION

Before creating a new file or subsystem, verify:

- Does this capability already exist?
- Is there an earlier proof of concept?
- Is there an existing file with the same responsibility?
- Can the existing implementation be reused?
- Would the proposal duplicate the architecture?
- Is the repository synchronized with the documented state?

If uncertain:

> STOP.
>
> Inspect the repository.
>
> Do not guess.
>
> Do not invent.

## FOUNDER WORKING PROTOCOL

Founder is not a programmer.

The CTO must:

- Explain the product state in simple Vietnamese.
- Tell Founder exactly which file to open or create.
- Provide full replacement content for short files.
- Build related files as one subsystem batch.
- Provide the exact command to run.
- State the expected result.
- Wait for GREEN before continuing.
- Avoid unnecessary technical explanations.
- Report product value, safety, remaining gaps, and future direction.

---

## SURVIVAL RULE

Updating the current organizational state is as important as updating Git.

A mission or work session is not properly closed until:

1. The required capability is GREEN.
2. `docs/OPERATING_CONTEXT.md` is updated.
3. `docs/CAPABILITY_CATALOG.md` is updated when capabilities changed.
4. Git changes are reviewed.
5. Changes are committed.
6. Changes are pushed.
7. The working tree is clean.

Git stores the history of the code.

Operating Context stores the current state of the organization.

Both must remain synchronized.
