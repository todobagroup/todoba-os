# ORGANIZATIONAL CONTINUITY

Version: 1.0

---

# PURPOSE

Organizational Continuity is the capability that allows TODOBA to continue operating without losing knowledge, context, mission, or direction.

The organization must never depend on a single CTO, a single worker, or a single machine.

Knowledge belongs to TODOBA.

---

# PRINCIPLE 1

The organization always knows where it is.

It always knows:

- Current Sprint
- Current Mission
- Current Priority
- Current Architecture
- Current Technical Debt

---

# PRINCIPLE 2

Every work session has a lifecycle.

BOOT

↓

MISSION

↓

IMPLEMENTATION

↓

VERIFICATION

↓

KNOWLEDGE UPDATE

↓

COMMIT

↓

PUSH

↓

CLOSE

No session is considered complete until PUSH has succeeded.

---

# PRINCIPLE 3

Knowledge must outlive people.

Every CTO is temporary.

Every Worker is replaceable.

Knowledge remains.

---

# PRINCIPLE 4

Context must never be reconstructed from chat history.

Context is generated by the organization itself.

The latest operational state must always be available.

---

# PRINCIPLE 5

Every completed session produces organizational knowledge.

Each session must update:

- Operating Context
- Decisions
- Lessons
- Technical Debt
- Next Mission

---

# PRINCIPLE 6

A new CTO must become productive within 30 minutes.

The organization is responsible for onboarding the CTO.

The CTO is not responsible for discovering organizational knowledge.

# KNOWLEDGE CURATOR

Purpose

Maintain organizational continuity.

Responsibilities

- Summarize completed work.
- Update operating context.
- Record lessons.
- Record decisions.
- Generate CTO Briefing.
- Prepare next mission.

Knowledge Curator does not write business logic.

Knowledge Curator maintains organizational knowledge.
